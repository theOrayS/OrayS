# OrayS 开发日志

本目录保存 OrayS 每个 PR 的人工可读开发过程记录，用于长期维护、问题复盘、比赛评审和团队交接。
日志必须在 PR 开始时创建并在开发过程中持续更新，不得在结束后凭记忆一次性补写。

## 目录结构

```text
docs/development-logs/
├── README.md
├── TEMPLATE.md
└── YYYY/
    └── MM/
        └── YYYY-MM-DD-pr-<number|draft>-<slug>.md
```

示例：

```text
docs/development-logs/2026/07/2026-07-15-pr-draft-unified-test-runner.md
docs/development-logs/2026/07/2026-07-18-pr-42-fix-mmap-cow.md
```

取得 PR 编号后，使用 `git mv` 将 `pr-draft` 改为正式编号。

## 一份日志必须回答的问题

1. 为什么做，目标和非目标是什么？
2. 开发前基线是什么，哪些失败是既有问题？
3. 采用了什么方案，为什么没有采用其他方案？
4. 开发过程中遇到了什么现象，如何定位根因并验证修复？
5. 哪些文件、符号和提交发生了变化？
6. 是否使用 AI；AI 影响了什么，人工做了哪些判断和修改，如何验证？
7. 是否参考或借鉴外部项目；来源、版本、许可证和改写范围是什么？
8. 最终执行了哪些测试；实际退出码、架构、镜像哈希和原始证据在哪里？
9. 尚有哪些限制、风险、后续工作和回滚方式？
10. 谁完成了独立复核，负责人是否能够在无 AI 辅助下解释和调试该改动？

## 记录粒度

- 按日期或 checkpoint 追加记录；一次记录应对应一个有意义的调查、修改或验证阶段。
- 保留失败和被否决方案；不要只记录最终成功路径。
- 记录可复现命令和退出码，不使用“测试正常”“应该通过”等模糊描述。
- 修复 flake 时保留第一次失败及诊断信息，不能用重试后的绿色覆盖它。
- 不粘贴超长控制台输出；只保留关键错误摘要和原始 artifact/run ID。

## 原始产物

大体积控制台日志、JSON/JUnit/HTML 报告、QEMU 输出、overlay 和临时镜像存放在：

```text
test/output/<run-id>/
```

该目录必须被 Git 忽略，或由 CI 作为 artifact 保存。开发日志只提交摘要、结果、run ID、哈希和相对证据路径。

## 隐私与安全

不得提交 token、SSH key、cookie、个人数据、无关主机路径或完整未脱敏 AI 对话。官方镜像本体不得进入 Git；只记录文件名、来源/版本和 SHA-256。

## 状态

日志 front matter 中使用：

```text
draft | blocked | ready | merged
```

只有两个架构的完整官方测试明确通过、独立复核完成且披露信息完整时，日志才能标记为 `ready`。
