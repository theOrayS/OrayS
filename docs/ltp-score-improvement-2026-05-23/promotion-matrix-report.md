# LTP promotion matrix stats/report lane

## Scope

Worker-4 stats/report lane inspected `scripts/ltp_summary.py` and the existing 2026-05-22 candidate reports. No production code change is needed for this lane: the existing summary tool already has a `--promotion-candidates` mode that combines multiple LA/RV logs into one required arch/libc promotion matrix.

This report records the reusable command recipe and a fresh current-output matrix generated from `output_la.md` + `output_rv.md`.

## Existing tool behavior

`scripts/ltp_summary.py` parses evaluator/LTP logs and classifies each case row by:

- architecture inferred from filename: `la`, `rv`, or `unknown`;
- libc inferred from test group name: `musl`, `glibc`, or `unknown`;
- wrapper status/code from `PASS LTP CASE`, `FAIL LTP CASE`, and `TIMEOUT LTP CASE`;
- internal LTP markers: `TFAIL`, `TBROK`, `TCONF`;
- timeout markers, excluding the harmless `Timeout per run` configuration line;
- `ENOSYS` / `not implemented` markers;
- panic/trap markers;
- runtime and memory cleanup deltas when emitted by the runner.

Promotion candidates require every requested arch/libc combo to be present and clean. Any non-PASS wrapper status, internal marker, timeout, ENOSYS, panic/trap, or missing LA/RV x musl/glibc combo blocks the case.

## Command recipe for future targeted LA/RV logs

Save targeted logs with architecture in the filename, for example:

- `docs/ltp-score-improvement-2026-05-23/rv-<batch>.log`
- `docs/ltp-score-improvement-2026-05-23/la-<batch>.log`

Then generate both human and machine-readable matrices:

```bash
python3 scripts/ltp_summary.py \
  --promotion-candidates \
  --promotion-arches rv,la \
  --promotion-libcs musl,glibc \
  output_la.md output_rv.md \
  docs/ltp-score-improvement-2026-05-23/rv-<batch>.log \
  docs/ltp-score-improvement-2026-05-23/la-<batch>.log \
  > docs/ltp-score-improvement-2026-05-23/<batch>-promotion-matrix.md

python3 scripts/ltp_summary.py \
  --promotion-candidates \
  --promotion-arches rv,la \
  --promotion-libcs musl,glibc \
  --json \
  output_la.md output_rv.md \
  docs/ltp-score-improvement-2026-05-23/rv-<batch>.log \
  docs/ltp-score-improvement-2026-05-23/la-<batch>.log \
  > docs/ltp-score-improvement-2026-05-23/<batch>-promotion-matrix.json
```

Use `output_la.md output_rv.md` in the input list when the batch is additive and should preserve current stable evidence. Omit them when auditing only a new targeted batch.

## Promotion rule for leader integration

A case should be proposed for stable-list promotion only if the generated matrix shows all four required combos:

- `la:glibc`
- `la:musl`
- `rv:glibc`
- `rv:musl`

and every row is free of:

- wrapper `FAIL` / `TIMEOUT` / unknown status;
- `TFAIL`, `TBROK`, `TCONF`;
- timeout markers;
- `ENOSYS` / not-implemented markers;
- panic/trap markers;
- event failures.

Exception: existing `read02` remains transparent stable evidence with `TCONF=2` per combo and must not be hidden or treated as a new clean promotion.
